# Part1
- $Compression\space Ratio = \frac{Uncompressed}{Compressed}>1$

- RLE (Run Length Encoding): 将字符串重新排列为字符 + 重复次数
	如果原文不多，反而没什么用甚至变长

- BWT (Burrows-Wheeler Transform):  所有“循环移位”进行排序，取字典排序后的所有末位排列。
	不直接压缩，排序后更加容易被RLE压缩，原理是相似的上下文前面经常出现相似甚至相同的字符

- 压缩的好处：
	1. 节省网络带宽
	2. 节省存储成本

- 意想不到的压缩应用场景：
	1. 软件更新
	2. 配置文件
	3. 数据库更新
	4. Web Service 返回的数据流
	5. JSON/XML 等结构化数据

- 压缩过的东西在用其他算法被压缩往往没有什么效果，因为压缩算法基本上都是利用原始数据的冗余，压缩过后基本上就是没有规律随机出现的字节了，没有压缩的空间了。

- 压缩可以用来`衡量相似度`，如果两个东西放在一起压缩，比分别单独压缩更省空间，说明两者拥有common pattern，更加相似

- 高效往往牺牲了可读性，`xxd`，用来读二进制文件的帮手，能将二进制文件转换成对应的十六进制数对。00000000  =>  48 65 6c 6c 6f

- 高效压缩有时候是lossy的，牺牲了精确还原的能力（但感觉其实可以通过预先设置规则后续部分复原）

- 如果每个类别的概率一样，那么就很难压缩，反之则相对容易压缩 

- `ASCII` 没有考虑字符的使用频率，假设所有字符都是一个使用频率，所以每个字符长度是一样的，理想来说，如果e比z使用频率更高，e的编码应该更短才合理

- `UTF-8` 考虑了这一点，是可变的，常见的英文字符是1bytes，而其他的可能是2bytes，中文好像是3bytes. 

- 可变长度固然香，但是有一个问题，怎么确定哪里是一个词的结束还是一个词的开始，这时候就需要遵守 `prefix-free code` 原则了。即，任何一个编码都不是其他编码的`前缀`。

- 不只压缩，还有检索，基于BWT的格式可以高效的搜索模式串，比如"San Jose"，这个在日后课程里的FM-index也会聊到

- BWT天然和字符串搜索有关。利用BWT这类结构,可以快速定位字符串开头的位置范围,所以压缩后的结构不一定妨碍搜索,有些压缩结构反而支持高效搜索。

# Part2
- source message/symbol  =>  codeword

- source message或者source symbol是字符串划分 (partitioned) 的基本单位。所以在编码前你要先决定你怎么将原文切成基本单位

- 然后有4种编码类型。 Block代表的是长度固定。 Variable代表的是长度可变。
	1. block to block。两个都是固定的长度。比如说ASCII码。
	2. Block to variable。比如说Huffman coding。常见字符给短编码，少见字符给长编码。
	3. Variable to block，比如说LZW。它会把不同长度的字符串片段编码成固定长度的编号。诶，你可以理解为，一个序号对应着一个token。这个token的长度是不固定的。
	4. variable to variable。比如说Arithmetic coding。

- 对称压缩指的就是解压时间和压缩时间差不多。非对称压缩一般指的是压缩非常慢但是解压必须得要快。前者的话用于视频会议场景，后者的话可能会用于软件安装包网页资源电子书等

- Decodable指的是每一个codeword都能和其他的进行区分，也就是一对一映射。
- Uniquely Decodable指的是这些C如果连在一起的话，仍然只有一种拆分方式。
- Prefix free code 一定是 Uniquely Decodable，但是Uniquely Decodable 不一定是 Prefix Free. 比如 {1, 100000, 00}

- 为什么喜欢prefix code因为它可以即时解码，只需要读到一个完整的C就能马上判断不用等后面更多的比特

- Static code指的是编码映射，在传输前就固定好了。霍夫曼编码就是如此。意思就是编码表，它不会变。这意味着你需要提前知道这些符号的出现的频率,常见就给他短编码，少见就给他长编码。优点是实现相对清楚，解码稳定。缺点是如果数据分布变化，固定编码可能就不是最优的了。

- Dynamic code指的就是mapping会随着时间变化。也叫adaptive coding。他这个动态指的是并非先固定文章的概率，而是一边读一边更新概率和mapping

- 如何评价压缩算法呢？看复杂度和压缩效果。

- measurement of information信息熵，越高越不确定，取值范围在$0-log_2(n)$ 
	$H=-\sum P(s)log_2P(s)$ 

- 香农商意味着最低的平均编码长度比如香农熵为2.1,那么就说明他们的平均编码长度最小必须得是2.1比特

## Huffman Encoding
- 霍夫曼树有两个特点，第一无损压缩，第二可变长度编码

- 霍夫曼树的步骤
	1. 找出概率/频率最低的两个符号
	2. 把它们合并成一个新节点
	3. 新节点的频率 = 两个频率之和
	4. 重复这个过程，直到只剩一个根节点
	5. 从根到叶子走路径，左边记 0，右边记 1

## Arithmetic Encoding
- 算术编码和霍夫曼编码最大的不同就是可以用小数比特来表示。

- 注意，小数比特指的是平均比特。算术编码将多个符号一起编码，然后取整，再将误差摊到整个消息上。

- 假设有三个符号：`A` 概率 0.5，`B` 概率 0.3，`END` 概率 0.2。
- 先把区间 `[0,1)` 按概率切成：`A = [0,0.5)`，`B = [0.5,0.8)`，`END = [0.8,1)`。
- 如果要编码 `ABEND`，初始区间是 `[0,1)`；读到 `A` 后缩小到 `[0,0.5)`；
    - $range = high - low$
    - $high = low + range * high(symbol)$
    - $low  = low + range * low(symbol)$
- 读到 `B` 后，在当前区间 `[0,0.5)` 内取 `B` 对应的比例范围，得到 `[0.25,0.40)`；
- 读到 `END` 后，在 `[0.25,0.40)` 内取 `END` 对应范围，得到最终区间 `[0.37,0.40)`。

- 虽然算术编码比霍夫曼编码的压缩率更高，但实际上通常用的更多的是霍夫曼编码。因为霍夫曼编码它更简单，实现更快，更加成熟，并且不用担心专利问题。
